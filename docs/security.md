# Security and trust boundaries

## Security posture

`qsio-kernel` is an experimental, in-process reference runtime. Its current controls help keep prototype semantics bounded, but they do not form a production security boundary.

## Assets

The kernel is intended to preserve:

- QSO identity and lifecycle state;
- transition preconditions and postconditions;
- provenance references;
- ordered interaction history;
- content-hash consistency; and
- the ability to reconstruct state through replay.

## Trust model

| Element | Current trust assumption |
| --- | --- |
| Caller | May provide malformed or forbidden QSI payloads |
| Python process | Trusted to execute the kernel code correctly |
| RuntimeContext | Mutable trusted state held in memory |
| Witness generator | Same process as the executor; not independent |
| Logical clock | Deterministic test/demo mechanism, not trusted wall time |
| Ledger | Ordered in-memory list; not durable or tamper-evident outside process memory |
| Content hashes | Integrity checks over canonical payloads; no authorship proof |

## Implemented safeguards

- Domain-separated content hashing for semantic records.
- Precondition hashes on state transitions.
- Structural QSIO verification.
- Rejection of transition requests containing `external_io`, `network`, `subprocess`, or `spawn` keys.
- Validation of known initiators and participants.
- Explicit genesis authorization flag.
- Quietus lifecycle blocking.
- Runtime limit records, including zero permitted spawned objects.
- Replay tests for final state consistency.

## Important limitations

### No sandbox

Forbidden-key validation is an application-level semantic check. It does not isolate Python code, restrict imports, prevent a caller from executing code elsewhere in the process, or enforce operating-system network/filesystem policy.

### No independent attestation

Witness records are generated by the same runtime that applies the transition. A verified witness flag currently means the local execution path marked it verified; it does not mean a separate authority observed the event.

### No durable tamper evidence

The ledger is an in-memory list. Parent hashes and content hashes support consistency checking, but an attacker controlling process memory can replace both records and state. Durable append-only storage is outside the current scope.

### Incomplete authorization enforcement

Capability and permission types exist, but the main interaction path does not yet require a capability for every operation. They must not be treated as a complete access-control system.

### Deterministic timestamps

Runtime timestamps are generated from a logical clock and a fixed date string. They provide ordering for the prototype, not externally verifiable time.

## Threats and current response

| Threat | Current response | Residual risk |
| --- | --- | --- |
| Request for external execution | Forbidden transition keys rejected | Alternate in-process code paths are not sandboxed |
| Unknown actor mutation | Initiator/participant registry validation | Registry ownership is process-local |
| Stale-state transition | Precondition hash verification | No concurrent transaction isolation |
| Ledger record corruption | Content-hash recomputation | No durable trusted copy |
| Mutation during Quietus | Lifecycle guard | Resume authority is process-local |
| False witness claim | QSIO requires verified witness for accepted record | Witness is not independent or signed |
| Replay divergence | Automated final-hash test | Limited to covered schemas and in-memory history |

## Safe operating guidance

- Run the kernel only with trusted Python code.
- Treat QSI payloads as untrusted input and validate them before deriving downstream effects.
- Do not attach real credentials, private keys, payment authority, production data, or external execution privileges.
- Do not expose the current runtime directly as a network service.
- Preserve test fixtures when changing canonicalization or hashing.
- Record any semantic change in the task chain, release plan, and changelog.

## Security gates for future capability

Before adding persistence, networking, signatures, external models, or federation, require:

1. a documented authority and threat model;
2. typed and versioned input/output schemas;
3. least-privilege capability enforcement;
4. adversarial and malformed-input tests;
5. independent verification or signature policy where claimed;
6. durable provenance and migration rules;
7. secret-handling and logging policy;
8. dependency and supply-chain checks;
9. incident containment and rollback procedures; and
10. explicit release approval.

## Reporting

Security findings should be described with the affected version, minimal reproduction, expected invariant, observed behavior, and potential impact. Do not include live credentials or sensitive third-party data in a public issue.
